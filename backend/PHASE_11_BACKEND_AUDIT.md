## 已发现并处理的后端边界问题

1. 提醒状态操作现在先确认提醒仍属于当前用户且来源仍有效，再改变状态。
2. 来源已经消失的提醒返回 404，不会出现“先修改数据库、再返回 404”。
3. 候选提醒对象不再通过 `pop` 改写内部候选数据。
4. 增加了店长门店范围、跨用户提醒 ID、失效提醒三类专项验证。
5. 根据前端 PR #6 反馈增加了分页前 `unread_only` 服务端筛选。

## 尚需契约负责人裁决的差异

1. 草案允许 trainer，后端候选实现和当前管理前端都排除 trainer。
2. 草案列出 admin/trainer/student 三条看板路径，候选实现只实现 admin。
3. 管理看板草案引用通用列表响应，候选实现返回具体对象。
4. 提醒候选实现返回分页对象，当前前端 adapter 期待扁平数组。
5. read/dismiss 成功后是否应返回更新后的提醒对象。
6. 提醒 GET 创建每用户状态记录是否可接受。
7. 来源消失再出现时是否继续沿用已读/忽略状态，以及状态保留期。
8. 分类、优先级、排序、分页默认值/上限和站内路径白名单。

## 交付门槛

后端候选实现提交后，下一步不是直接改前端，而是：

1. 契约负责人评审并冻结上述差异；
2. 后端按冻结契约做最后一次收敛并更新 API 状态；
3. 前端重新生成类型，更新 adapter、Mock 和测试；
4. 双方使用真实 API 覆盖空数据、分页、401、403、404、跨公司、跨门店和跨用户；
5. 真实联调与浏览器 E2E 均通过后，才更新集成与验证状态。
