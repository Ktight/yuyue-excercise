# 瑜悦练商业化视觉方案生成提示词

## 使用方式

将本文件连同项目仓库、`页面.md` 和最新 UI 参考文件一起交给负责视觉方案的设计师或 AI。第一轮只制作代表页面和主题系统，老板确认方向后再扩展到全部页面。

---

## 可直接复制的提示词

你是一名资深商业产品设计师、UI 设计总监和前端视觉工程师。请基于“瑜悦练”现有项目，设计并实现多套可在浏览器中直接预览、对比和汇报的商业级前端视觉方案，供公司负责人确认最终产品风格。

这不是概念插画任务，也不是随意制作几个漂亮页面。方案必须基于项目真实存在的页面、功能和内容，具备成熟商业产品的视觉品质，能够落地到 Vue 3 前端，并保持现有功能、路由、字段、权限和交互逻辑不变。

### 1. 项目背景

- 产品名称：瑜悦练
- 产品类型：瑜伽馆经营与训练服务系统
- 已存在角色：超级管理员、公司管理员、门店管理员、训练师、学员
- 技术栈：Vue 3、TypeScript、Vite
- 前端采用功能模块、后端适配层和集中式样式分层
- 后端尚未完全完成，部分页面使用 Mock；视觉设计不得依赖后端先完成
- `页面.md` 是当前页面数量和功能范围的事实基线

### 2. 开始前必须审查

完整检查以下内容，并以代码事实为准：

1. `页面.md`
2. 全部前端路由配置
3. 全部页面级 Vue 组件
4. `frontend/src/features` 下的功能模块
5. `frontend/src/app/components` 下的公共组件
6. `frontend/src/shared` 下的公共能力
7. `frontend/src/app/styles` 下的样式层、设计变量和主题说明
8. 已有测试、Mock、开发计划和进度记录
9. `contracts/openapi.yaml`、`contracts/enums.json`（只读）
10. 最新的 `yoga-app-ui-guide (1).html`（如存在）

禁止凭空增加代码、路由和文档中不存在的业务功能。规划中但尚未实现的内容必须明确标记，不得伪装成当前可用功能。

### 3. 视觉目标

生成三套差异明显、但都达到商业产品质量的视觉方向。三套方案不能只是换主色，必须在品牌气质、字体、色彩、留白、卡片、导航、图片、信息密度、表单、圆角、阴影、动效和数据呈现上形成系统差异。

#### 方案 A：自然疗愈与高级生活方式

关键词：自然、呼吸感、柔和、松弛、高级、温暖、东方审美、精品瑜伽工作室。

- 推荐米白、暖灰、鼠尾草绿和低饱和植物色
- 使用克制阴影、较大留白和高品质瑜伽摄影
- 女性友好但不过度甜美
- 避免廉价粉色、装饰堆积和泛滥渐变

#### 方案 B：现代运动与专业训练

关键词：专业、高效、清晰、活力、可信赖、数据化、训练计划感。

- 使用清爽中性色和明确的品牌强调色
- 强化信息层级、状态和数据表达
- 紧凑但不拥挤，操作入口明确
- 避免健身房式粗暴视觉和过度荧光色

#### 方案 C：轻奢健康与数字会员服务

关键词：精致、高端、会员感、品牌化、克制、私人健康服务。

- 推荐深墨色、象牙白、香槟色或低饱和金属色
- 使用精致字体比例、细线分隔和克制动效
- 强调内容品质和会员身份
- 避免大面积金色、伪奢华和过度装饰

如果认为某个方向不适合现有产品，可以提出更合理的替代方向，但必须说明商业依据。

### 4. 第一轮代表页面

不要在老板确认前把全部页面重复制作三遍。第一轮从 `页面.md` 中制作以下代表模板：

1. 登录页：验证品牌第一印象和认证表单
2. 管理员工作台：验证导航、信息架构和经营数据表达；当前功能是占位，展示内容不得冒充已实现功能
3. 用户管理列表：验证搜索、筛选、列表、分页和状态反馈
4. 新建学员：验证复杂业务表单
5. 学员详情：验证档案和多标签信息结构
6. 门店详情：验证详情、编辑、状态操作、场地列表和内嵌表单
7. 课程模板列表或详情：验证内容卡片、标签和状态切换
8. 403/404：验证系统状态页

复用同一页面模板的路由不应重复制作。占位模块不得自行补造业务数据。

### 5. 必须覆盖的状态

按照页面实际适用范围展示：

- 正常内容、加载、空数据、网络错误和重试
- 表单默认、聚焦、校验错误、禁用、提交中、提交成功和提交失败
- 未登录、无权限和页面不存在
- 激活、停用等业务状态
- 桌面端和移动端窄屏状态

状态必须进入统一组件体系，不能在不同页面中重复设计成互不一致的形式。

### 6. 商业化审美标准

- 第一眼能够识别为瑜伽、健康或运动服务产品
- 三套方向第一眼即可区分
- 不得呈现为通用后台模板或 AI 渐变落地页
- 不得大量使用无意义卡片、阴影、胶囊和装饰图形
- 不使用表情符号代替正式图标
- 不使用无授权商业图片
- 不牺牲可读性追求所谓高级感
- 使用真实、克制、可信的商业文案
- 满足基本对比度、焦点可见性和键盘操作要求
- 移动端需要重新组织信息层级，不能只缩小桌面版

### 7. 前端实现边界

视觉实现必须与业务逻辑解耦：

- 三套方案通过主题变量切换，不复制三套业务页面
- 不修改接口、适配器、状态管理、路由、权限、表单字段和校验规则
- 不修改 `backend/**`、`contracts/**`
- 不把主题判断散落到页面组件
- 不在 Vue 页面堆积重复颜色值
- 新组件优先使用语义变量；旧页面按触达原则逐步迁移
- 保留用户已有修改，不进行与视觉无关的重构

主题层建议保持：

```text
reset → tokens → theme → base → layout → components → utilities
```

语义变量至少覆盖页面与卡片背景、各级文字、品牌操作色、危险色、成功色、边框、焦点环、字体、字号、圆角、阴影、间距、控件高度、页面宽度和动效。

### 8. 评审主题切换器

提供一个可移除的视觉评审入口，在同一份前端中切换：

- 方案 A：自然疗愈
- 方案 B：专业训练
- 方案 C：轻奢会员

切换后功能和数据必须保持不变，主题立即生效，显示当前方案名称并记住最近选择。评审能力不能侵入正式业务逻辑，最终定稿后应能整体移除。

### 9. 对比报告

每套方案说明：一句话定位、目标感受、品牌关键词、色彩、字体、图片、卡片、表单、导航、动效、适合场景、商业优势、风险、实现成本和长期维护成本。

提供以下横向比较：

| 实现成本 | | | |

可以给出专业推荐，但不得替负责人做决定。推荐必须依据目标用户、产品定位、功能结构和维护成本。

### 10. 最终交付

1. 对 `页面.md` 的核查结果和差异说明
2. 三套视觉方向定义
3. 三套设计变量
4. 代表页面的可运行实现
5. 主题切换评审入口
6. 桌面端与移动端预览
7. 核心状态预览
8. 公共组件展示页
9. 三套方案横向比较报告
10. 最终方案扩展到全部页面的工作量说明
11. 尚未实现或无法确认的内容清单
12. 负责人确认表

负责人确认表至少包括总体方案、主色、字体、圆角、信息密度、图片风格、导航形式、特殊页面调整、确认人和确认日期。

### 11. 执行顺序

1. 审查代码、文档和 `页面.md`
2. 确认真实页面范围
3. 提交三套风格定义和商业依据
4. 选择代表页面
5. 建立主题变量和切换能力
6. 实现代表页面
7. 补齐关键状态
8. 检查桌面端、移动端和无障碍
9. 运行 lint、typecheck、测试和 build
10. 生成对比报告并等待负责人确认
11. 确认后才扩展到全部页面

如果视觉改造会影响业务逻辑、接口适配、路由、权限或契约，立即停止该部分修改并说明冲突，不得为了视觉效果改变功能。
