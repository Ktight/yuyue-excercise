# 瑜悦练前端生成思路、设计思路与前后端协作说明

更新时间：2026-07-21  
适用项目：瑜悦练（`Ktight/yuyue-excercise`）  
当前前端分支：`frontend/phase-2-auth-integration`

## 1. 文档目的

这份文档用于向后端开发者说明前端是怎样规划、生成、实现和验收的，以及前端需要从后端获得哪些稳定信息。

主要目标不是要求后端服从前端代码结构，而是消除以下信息差：

- 后端不知道前端依据什么契约生成类型和 Mock；
- 前端不知道某个接口是未实现、暂缓实现，还是已经可以联调；
- 同一个字段在契约、后端实现、Mock 和页面中含义不同；
- 双方都认为某项功能已经完成，但完成标准并不一致；
- 后端变更后，前端不知道应该修改哪个位置；
- AI 辅助开发时，因为上下文不完整而猜测字段或提前实现后续功能。

我们希望建立的共同理解是：

> 后端负责真实业务规则、数据和运行接口；契约负责描述双方共同认可的边界；前端负责自己的领域模型、交互和适配层。任何一方都不通过猜测替另一方补全信息。

## 2. 为什么前端要先做详细功能拆分

本项目大量使用 AI 辅助开发。如果只给 AI 一份很长的总设计方案，容易出现以下问题：

- 一次跨多个模块修改；
- 同一个功能散落在多个页面中；
- 为了完成当前步骤，提前实现后续 Phase；
- Mock 字段由 AI 自行猜测；
- 页面直接调用 HTTP 客户端；
- 一个字段变化导致到处搜索和修改；
- AI 看不到完整上下文时重复创建相似工具；
- 功能看似完成，但没有明确的验收命令和状态记录。

因此前端采用“总方案 → Phase → 功能文件 → 单步骤任务”的逐层拆分方式。

```text
产品与界面设计
    ↓
前端总体实施方案
    ↓
Phase 阶段划分
    ↓
一个功能一份详细说明
    ↓
一个可复制的 AI 实现指令
    ↓
只实现一个可验收步骤
    ↓
测试、构建、记录和交接
```

每个功能说明应尽量独立。AI 完整读取该文件后，应当能够理解：

- 功能目标；
- 前置条件；
- 允许修改的目录；
- 禁止修改的内容；
- 前端领域模型；
- 所需后端能力；
- Mock 方案；
- 组件和文件划分；
- 状态和错误处理；
- 测试要求；
- 验收条件；
- 后续功能边界。

这种拆分不是为了制造更多文档，而是为了让每次 AI 开发都有明确边界，减少上下文丢失和跨模块修改。

## 3. 前端的分层设计

### 3.1 分层原则

前端代码按职责分层：

```text
页面与布局
    ↓
功能组件
    ↓
状态与前端领域模型
    ↓
后端适配层
    ↓
统一 HTTP 客户端
    ↓
后端 API
```

各层职责如下。

#### 页面和布局

负责：

- 页面结构；
- 路由组装；
- 加载、空状态、错误状态；
- 调用功能用例或 Store；
- 展示前端领域对象。

页面不负责：

- 拼接后端 URL；
- 解析响应包裹；
- 处理 snake_case；
- 存放 Mock 数据；
- 猜测错误码。

#### 功能组件

负责：

- 表单输入；
- 用户交互；
- 展示数据；
- 前端即时校验；
- 通过 props/emits 与外部通信。

组件不直接访问后端，不直接读取 OpenAPI 生成类型。

#### Store 与领域模型

负责：

- 前端需要维护的业务状态；
- 前端稳定使用的 camelCase 模型；
- 会话生命周期；
- 跨组件共享状态。

这里使用的是“前端如何理解数据”，不是“后端 JSON 原样长什么样”。

#### 后端适配层

路径：

```text
frontend/src/backend-adapters/
```

负责：

- 后端路径；
- HTTP 方法；
- 请求体字段；
- 查询参数；
- 统一响应包裹；
- snake_case 与 camelCase 转换；
- OpenAPI 生成类型与前端领域模型的转换；
- 后端兼容性变化。

这是前端唯一需要理解后端传输格式的业务层。

一个业务域对应一个适配文件，例如：

```text
auth.adapter.ts
users.adapter.ts
```

如果后端修改登录响应字段，前端优先只需要检查 `auth.adapter.ts`，而不需要进入登录页面、Pinia Store 和布局组件逐个修改。

#### 统一 HTTP 客户端

负责：

- API Base URL；
- Authorization 请求头；
- 网络错误转换；
- 统一错误响应转换；
- 通用 Axios 配置。

它不处理具体业务字段，也不知道什么是用户、课程或门店。

## 4. 契约在前端中的作用

根目录契约是前后端共享的接口说明：

```text
contracts/openapi.yaml
contracts/enums.json
contracts/status.json
```

它们在前端中的职责不同。

### 4.1 `openapi.yaml`

用于生成请求、响应和 Schema 的 TypeScript 类型。

前端不手工复制一份后端 DTO，而是执行生成命令：

```powershell
npm.cmd run generate:api-types
```

生成结果位于：

```text
frontend/src/generated/api-types.ts
```

生成文件不手工编辑。契约变化后重新生成，让 TypeScript 找出不兼容位置。

### 4.2 `enums.json`

作为跨端枚举的唯一来源，例如：

- 用户角色；
- 资源状态；
- 性别；
- 课程分类；
- 出勤状态。

前端执行：

```powershell
npm.cmd run generate:enums
```

生成：

```text
frontend/src/generated/enums.ts
```

这样可以避免前端和后端分别维护一套枚举值。

### 4.3 `status.json`

用于判断接口是否可以真实联调，而不是判断 OpenAPI 中是否已经出现该路径。

建议共同使用的状态含义：

| 状态 | 前端处理方式 |
|---|---|
| `NOT_STARTED` | 只允许规划，不声称存在真实 API |
| `DRAFT` | 可以设计和 Mock，但不能当作稳定接口 |
| `IN_PROGRESS` | 可以提前适配，必须保留变化预期 |
| `API_READY` | 可以接入真实接口并开展联调 |
| `VERIFIED` | 前后端在约定环境完成验收后才能标记 |

如果一个模块是 DRAFT，但其中部分端点标记为 `API_READY`，双方需要明确联调准入到底按模块还是按单端点判断。前端不会自行猜测。

## 5. 前端如何处理后端尚未完成的功能

当后端接口尚未完成时，前端不会停止全部开发，也不会在页面中伪造数据，而是采用 MSW Mock。

Mock 的要求：

- 路径和方法与当前契约一致；
- 请求体与契约一致；
- 成功响应与统一包裹一致；
- 错误响应与错误码约定一致；
- Mock 数据放在功能的 `mocks` 目录；
- 页面不知道当前收到的是真实响应还是 Mock；
- 后端接口就绪后，关闭 Mock 即可切换到真实 API。

Mock 的意义是让前端独立验证自己的状态、交互和适配逻辑，不代表后端已经实现，也不代表前后端已经联调成功。

## 6. 后端依赖的记录方式

每个前端功能在实现过程中都需要识别后端依赖，例如：

- 缺少接口；
- Schema 未冻结；
- 枚举未冻结；
- 权限范围不明确；
- 错误码未定义；
- 分页规则未定义；
- 字段是否可空不明确；
- 功能明确安排在后续 Phase。

记录时必须区分：

### A. 当前代码确实未找到

表示我们检查了当前分支的路由和实现，没有发现契约中的端点。

这不自动等于 Bug，还要结合契约状态和 Phase 计划判断。

### B. 明确安排在后续 Phase

表示依赖模块尚未开始，当前阶段不具备完整实现条件。

例如门店真实性和跨公司门店归属校验，需要 Companies/Stores 模块完成后再补充。

### C. 需要对方确认

表示现有资料不足以判断范围，双方必须确认，不能由 AI 或某一端自行决定。

例如契约中有 DRAFT Logout，但 Phase 2 路由中没有 Logout，需要确认它属于当前 Phase 还是以后实现。

### D. 已具备联调条件

表示：

- 端点状态允许联调；
- 路径和方法确定；
- 请求/响应 Schema 确定；
- 权限确定；
- 错误结构确定；
- 后端可运行；
- 有可使用的测试账号或种子数据。

## 7. 一个前端功能的标准生成过程

### Step 1：读取功能说明

AI 在修改代码之前，需要完整阅读：

- 当前功能文件；
- 工程基础规则；
- 双人开发交接规范；
- 前端进度记录；
- 后端依赖记录；
- 当前契约及状态。

### Step 2：检查前置条件

确认：

- 前一个步骤是否完成；
- 当前分支是否正确；
- 工作区是否存在未处理修改；
- 依赖端点是否可以联调；
- 缺失内容是否可以安全 Mock。

### Step 3：声明修改边界

一次只实现一个功能或一个可验收步骤。

如果完成当前步骤会迫使 AI 同时修改多个业务模块，应停止并重新拆分，而不是继续堆叠代码。

### Step 4：先建立领域模型和接口边界

先明确页面需要什么前端数据，再由适配器把后端 DTO 转换为该模型。

不让后端字段命名直接扩散到所有组件。

### Step 5：独立实现组件和状态

尽量让每个能力有职责明确的文件，并通过该功能的 `index.ts` 输出。

组件只依赖公开入口，避免跨功能深层导入。

### Step 6：使用契约 Mock 验证

在真实后端未就绪时，通过 MSW 验证：

- 成功流程；
- 加载状态；
- 空状态；
- 字段错误；
- 权限错误；
- 网络错误；
- 重复提交；
- Token 失效等边界情况。

### Step 7：运行质量门禁

前端当前使用：

```powershell
npm.cmd run check:all
```

它包含：

- TypeScript 类型检查；
- ESLint；
- Prettier；
- 模块边界检查；
- 单元测试；
- 生产构建。

未经验证不能把功能标记为 `VERIFIED`。

### Step 8：更新记录并交接

记录：

- 修改文件；
- 完成内容；
- 测试命令和结果；
- 当前使用 Mock 还是真实 API；
- 新增或变化的后端依赖；
- 未完成事项；
- 下一步入口。

## 8. Phase 完成的共同定义

“前端 Phase 完成”不等于“前后端整体完成”。建议分别记录以下状态：

### 前端实现完成

- 页面、组件和状态实现完成；
- 适配器按当前契约完成；
- Mock 场景完成；
- 前端质量门禁通过。

### 后端实现完成

- 端点、权限和业务规则实现完成；
- 后端测试通过；
- 契约状态已更新；
- 可提供联调环境和账号。

### 联调完成

- 前端关闭对应 Mock；
- 使用真实后端完成核心流程；
- 请求、响应、错误和权限一致；
- 双方确认没有未记录的兼容处理。

### 验收完成

- 双方共同认可验收条件；
- 相关记录更新为 VERIFIED；
- 变更进入约定的集成分支或主分支。

任何一方单独完成自己的代码，都不应直接把“联调完成”或“验收完成”标记为真。

## 9. 前端希望后端在交接时提供的信息

每个端点达到可联调状态时，希望后端提供：

1. 方法和完整路径；
2. 当前契约状态；
3. 请求 Schema；
4. 成功响应 Schema；
5. 所有预期错误码和 HTTP 状态；
6. 字段级错误格式；
7. 允许访问的角色；
8. 租户和数据范围规则；
9. 分页、筛选和排序规则；
10. 是否存在幂等、Token 轮换等特殊行为；
11. 测试账号或种子数据创建方式；
12. 尚未实现但已出现在契约中的端点；
13. 明确留到后续 Phase 的业务规则；
14. 本次变更是否要求前端重新生成类型或枚举。

如果这些信息已经准确写入契约、状态文件和 README，后端只需告诉前端对应提交哈希，不需要重复人工整理。

## 10. 前端交接给后端的信息

前端完成一个功能后会提供：

1. 前端分支；
2. 提交哈希；
3. 已适配的端点；
4. 当前使用 Mock 还是真实 API；
5. 前端领域模型与后端 DTO 的映射位置；
6. 测试和构建结果；
7. 发现的契约疑问；
8. 当前代码确实未找到的后端端点；
9. 已知安排在后续 Phase 的依赖；
10. 联调时需要后端确认的事项。

前端不会要求后端理解页面内部组件结构，也不会要求后端为了前端展示结构改变领域设计。

## 11. 双方变更流程

### 后端或契约发生变化时

推荐顺序：

```text
后端提出业务变化
    ↓
双方确认接口影响
    ↓
更新 contracts 源文件和状态
    ↓
生成或更新 openapi.yaml / enums.json
    ↓
后端实现并测试
    ↓
前端重新生成 TypeScript 类型
    ↓
前端修改对应 backend adapter
    ↓
关闭对应 Mock 进行联调
```

不要先修改真实接口、最后才通知前端补契约，因为这会让契约失去共同边界的作用。

### 前端发现不匹配时

前端应提供可核对的事实：

- 请求方法和路径；
- 实际请求体；
- 实际 HTTP 状态；
- 实际响应；
- 契约期望；
- 前端适配器位置；
- 是否能在 Mock 中复现。

在缺少证据时，不直接称为后端 Bug。

## 14. 最终协作原则

- 契约是共同边界，不是某一端的内部文档；
- 状态文件必须反映真实进度；
- DRAFT 可以用于设计和 Mock，但不能冒充联调完成；
- 后端原始格式只能进入前端适配层；
- 页面和组件不猜接口字段；
- Mock 不代表后端实现完成；
- 单端完成不代表联调完成；
- 发现差异时先提供证据，再判断归属；
- 每次只推进边界清晰、可以独立验收的步骤；
- 所有跨 Phase 的依赖必须被明确记录，不能依赖口头记忆。

这套方式的目的，是让前端和后端都能独立推进，同时在真正接合时只处理少量、明确、可定位的差异，而不是在大量页面和接口中反复排查。
