# 前端 Phase 开发与完成标准

更新时间：2026-07-21  
适用范围：瑜悦练前端全部 Phase  
目的：让前端能够独立开发，把前后端变化限制在边界层，并用可复核证据判断一个 Phase 是否真正完成。

## 1. Phase 1/2 现有代码逻辑

### Phase 1：工程和前端公共基础

Phase 1 提供后续业务模块共同依赖的运行基础：

- `src/app/`：应用启动、路由和全局装配入口；
- `src/layouts/`：页面框架；
- `src/shared/`：HTTP、配置、Mock、测试工具等无业务归属能力；
- `src/components/`：跨业务复用组件；
- `src/generated/`：由 `contracts/openapi.yaml` 和 `contracts/enums.json` 生成，只作为契约输入；
- `scripts/check-boundaries.mjs`：检查模块依赖边界；
- `check:all`：依次执行生成、类型检查、Lint、格式检查、边界检查、单元测试和生产构建。

业务功能不得反向写入公共层。`shared/` 不得依赖 `features/`，不同 Feature 之间只能通过公开 `index.ts` 使用能力。

### Phase 2：认证和用户管理

当前请求链路是：

```text
页面/组件
  → Feature Store 或用例函数
  → features/<feature>/api 兼容出口
  → backend-adapters/<feature>.adapter.ts
  → shared/api/http-client.ts
  → 后端接口
```

响应返回时按相反方向流动。后端的路径、请求方法、响应包裹和 `snake_case` 字段只允许在适配器及契约型 Mock 中出现；页面、组件和 Store 使用前端领域模型。

Auth 当前逻辑：

1. 登录表单校验输入后调用 Auth Store；
2. Store 通过 Auth 适配器登录，并保存 Access/Refresh Token 与当前用户；
3. 应用恢复时读取本地 Access Token，再请求当前用户；
4. HTTP 层遇到并发 401 时共享一次会话恢复任务，刷新成功后重放请求；
5. Refresh Token 无效时清理会话，防止无限刷新；
6. 修改密码有独立页面、表单和 Schema，但真实提交仍需最终联调。

Users 当前逻辑：

1. 用户列表页维护 URL 查询、角色/关键字过滤和分页状态；
2. Users 适配器将后端分页及用户字段转换为前端模型；
3. 创建用户使用独立页面、表单和 Schema；
4. MSW 按契约模拟分页、创建和业务错误；
5. 真实用户列表已验证，真实创建用户及公司/门店关系仍需最终联调。

### 当前准确状态

- Phase 1：公共基础主体完成；仍应按功能清单确认未完成的公共组件，例如 `AppImageUploader`。
- Phase 2：前端实现基本完成，登录、当前用户、Token 刷新、用户列表已完成真实联调；其余项目见 `FRONTEND_STATUS.md` 的最终测试清单。
- Phase 3～13：契约当前标记为 `DRAFT/NOT_STARTED`，不得因为已有页面、Mock 或占位代码而记录为完成。

## 2. 前端独立与后端接合的分层标准

每个业务 Feature 使用以下职责边界：

```text
features/<feature>/
├─ pages/          路由页面，只负责页面编排
├─ components/     展示与交互组件
├─ model/          前端领域类型、Schema、Store、用例状态
├─ api/            Feature 稳定出口，不处理后端字段转换
├─ mocks/          契约一致的 MSW 场景
├─ routes/         本功能路由
└─ index.ts        唯一公开索引

backend-adapters/
└─ <feature>.adapter.ts  后端路径、方法、字段和响应转换的唯一位置
```

必须遵守：

- 页面和组件不能直接调用 `httpClient`；
- 页面、组件和 Store 不能导入 OpenAPI 生成类型；
- 后端 DTO 与前端领域模型不得共用同一个类型；
- Mock 数据不能写入 Vue 页面；
- 一个后端字段变化原则上只修改对应适配器、契约 Mock 和适配器测试；
- 适配器通过 `backend-adapters/index.ts` 索引，Feature 通过自身 `index.ts` 索引；
- 后端未完成时可以按已冻结契约使用 MSW，但必须记录依赖，不能猜字段；
- 契约未冻结且无法安全 Mock 时暂停该接合点，前端其余纯交互能力可以继续。

## 3. 一个完整 Phase 的交付物

一个 Phase 必须同时交付以下内容，缺一项只能记录为进行中：

1. **范围**：明确包含、不包含、前置依赖和跨 Phase 依赖；
2. **功能索引**：每个功能有独立目录、独立入口和可定位文件；
3. **领域模型**：前端语义稳定，不暴露后端传输格式；
4. **交互实现**：加载、空数据、成功、失败、重试、禁用和权限状态完整；
5. **适配器**：真实路径、方法、请求、响应、分页和错误转换集中处理；
6. **契约 Mock**：正常、边界、权限、校验和服务异常场景可重复运行；
7. **自动化测试**：Schema、Store/用例、适配器、关键组件与关键页面流程有测试；
8. **真实联调**：所有本 Phase `API_READY` 端点至少完成正常及关键异常验证；
9. **质量证据**：`check:all` 全部通过并记录日期与提交号；
10. **文档记录**：进度、后端依赖、已知问题、决策和遗留事项同步更新。

## 4. 商业级完成门禁（Definition of Done）

只有下列门禁全部满足，Phase 才能标记为 `VERIFIED`：

### 4.1 产品与业务

- 验收条件可测试且全部通过；
- 正常流程、空状态、边界值、重复操作和失败恢复均有定义；
- 角色、公司、门店及数据归属规则与产品约定一致；
- 金额、时间、时区、手机号、状态转换等关键字段不存在隐式假设。

### 4.2 架构与可维护性

- 每个功能职责单一并由 `index.ts` 暴露；
- 页面不理解后端 DTO，后端变化被适配层吸收；
- 没有跨 Feature 深层导入、循环依赖或复制粘贴的业务规则；
- 不以临时 `any`、硬编码 ID、静默降级或页面内兼容作为完成手段；
- 关键设计决策和暂时限制有记录。

### 4.3 正确性与测试

- TypeScript、ESLint、Prettier、边界检查、单元测试和生产构建全部通过；
- 关键用户旅程有端到端测试或可复现的人工联调记录；
- 权限拒绝、401/403、业务校验、网络失败和并发请求得到验证；
- Mock 与真实 API 的响应形状、错误码和权限结果完成差异核对；
- 不使用测试通过替代真实联调，也不使用真实联调替代自动化回归。

### 4.4 安全、性能与可观测性

- 敏感信息不写入源码、日志、URL 或测试截图；
- Token 生命周期、退出、失效和越权行为经过验证；
- 列表具备分页或明确的数据量限制，提交按钮防重复提交；
- 关键错误对用户可理解，对排查人员保留 `request_id` 等证据；
- 不在前端伪造后端权限，前端权限控制只负责交互，后端负责最终授权。

### 4.5 交付与回滚

- 变更只位于允许目录，未夹带后端、契约或无关文件；
- 提交和 PR 能说明范围、测试结果、真实/Mock 状态及遗留问题；
- 数据迁移、破坏性契约变化或功能开关有明确发布顺序；
- 出现故障时能够回滚前端版本，且旧版本与当前后端兼容范围已知。

## 5. 状态定义

统一使用以下状态，禁止用“页面看起来完成”代替真实状态：

| 状态 | 含义 |
| --- | --- |
| `NOT_STARTED` | 尚未开始 |
| `IN_PROGRESS` | 正在实现，尚不能完整验收 |
| `UI_READY` | 前端交互与 Mock 已完成，真实 API 尚未验证 |
| `API_READY` | 后端声明端点可联调，不代表前端已通过 |
| `INTEGRATED` | 指定端点或场景已完成真实联调 |
| `BLOCKED` | 有明确外部依赖，已记录证据与责任边界 |
| `VERIFIED` | 本 Phase 全部门禁通过并保存证据 |

状态必须落到具体功能或端点。部分端点 `INTEGRATED` 时，整个 Phase 仍可以是 `IN_PROGRESS`，不得向上扩大完成结论。

## 6. 后端依赖与问题记录模板

每个问题单独一条，避免把多个事实合并成无法关闭的大问题：

```markdown
### BE-<Phase>-<序号> 标题

- 发现日期：YYYY-MM-DD
- 关联前端功能：
- 前端提交号：
- 后端提交号/环境：
- 契约位置与状态：
- 请求方法和路径：
- 前置角色、公司、门店：
- 实际 HTTP 状态和响应摘要：
- 契约期望：
- 前端适配器位置：
- Mock 是否可复现：是/否
- 分类：后端未实现 / 后端偏离契约 / 契约待确认 / 后续 Phase 依赖 / 前端缺陷
- 当前处理：继续使用 Mock / 暂停接合点 / 已修复待验证
- 关闭证据：
```

没有真实请求、契约对照或可复现步骤时，只能记录为“待确认”，不能直接判定后端 Bug。

## 7. 每个 Phase 的标准执行顺序

1. 冻结范围和验收条件；
2. 检查契约状态与前置 Phase；
3. 建立 Feature 文件结构、领域模型和公开索引；
4. 先用契约 Mock 完成交互与异常路径；
5. 编写适配器和自动化测试；
6. 运行 `npm.cmd run check:all`；
7. 对 `API_READY` 端点执行真实联调；
8. 登记差异并只在边界层消化已确认变化；
9. 完成角色、租户、安全和回归测试；
10. 更新状态与证据，满足全部门禁后才能标记 `VERIFIED`。
